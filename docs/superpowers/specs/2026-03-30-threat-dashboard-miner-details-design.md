# Threat Dashboard Miner Details Design

## Summary

Extend the existing threat dashboard so each threat row can open a separate detail panel that shows the full red miner prompt, the blue miner's classification for that prompt, and miner-specific metadata for both sides. The current dashboard remains table-first, but gains a drill-down inspector that makes each threat event explainable without overcrowding the main list.

## Goals

- Keep the existing threat table fast to scan.
- Let operators inspect the exact prompt generated by the red miner for a threat event.
- Show the blue miner's classification result for that same prompt in the detail view.
- Include concise red and blue miner details alongside the selected threat.
- Preserve a usable experience on desktop and smaller screens.

## Non-Goals

- Building a new dedicated threat details route.
- Replacing the overall dashboard page layout.
- Wiring the dashboard to a live backend data source in this iteration if the page remains mock-backed.
- Adding deeper validator or judge transcript history beyond the selected threat's core details.

## Recommended Approach

Use a row-selected side panel inspector attached to the existing dashboard page. The table continues to show compact threat summaries, while selecting a row reveals full event details in a companion panel. This keeps the current information density under control and avoids interruptive modal behavior.

## Architecture

### Data Model

Extend the dashboard threat entry shape in `frontend/data/mock.ts` so each threat record contains both the compact list data and the full inspector payload. Each selected threat should carry:

- the full red miner prompt text
- the blue miner classification verdict tied to that prompt
- concise red miner metadata
- concise blue miner metadata

The table should continue to consume a single threat collection, but the records become richer so the selected row can drive the inspector without extra lookup logic.

### Component Boundaries

Keep `frontend/components/dashboard/ThreatTable.tsx` responsible for the tabular overview and row selection. Add a dedicated detail component, such as `frontend/components/dashboard/ThreatDetailsPanel.tsx`, that renders the selected threat's full content.

The dashboard page in `frontend/app/dashboard/page.tsx` should own the active-selection state and coordinate the two child components:

- `ThreatTable`
  renders the list, exposes row click handling, and highlights the active row
- `ThreatDetailsPanel`
  renders the selected event's prompt, classification, and miner details

This keeps the table readable and isolates the richer event presentation in a focused component.

## UI Design

### Table Behavior

The threat table should continue showing the compact columns already present:

- timestamp
- payload preview
- attack type
- threat score
- status

Rows become interactive. Clicking a row marks it as selected and updates the inspector panel. The selected state should be visually distinct but still consistent with the existing dashboard styling.

### Inspector Panel

The new detail panel should sit beside the table on larger screens and behave like a slide-over panel on narrower screens. It should display:

- the full red miner prompt
- the blue miner's classification result
- red miner details such as uid, rank, severity, novelty, or similar scoring metadata
- blue miner details such as uid, rank, precision, recall, latency, or similar defense metadata

The classification result should read as the outcome for the selected prompt, not as a generic miner-wide status. The panel should stay compact enough to scan quickly, using grouped sections rather than long free-form text blocks.

### Default State

The page should open with a meaningful selection so the inspector is useful immediately. The simplest default is the newest threat entry in the existing list. If no threat entries exist, the panel should render a light empty state rather than collapsing the layout.

## Data Mapping

The current threat stream items already represent individual threat events. Each entry should be expanded to include:

- `redPrompt`
- `blueClassification`
- `redMiner`
- `blueMiner`

If the existing `payload` field represents the same prompt shown in the table preview, the inspector can treat `redPrompt` as the canonical full-text version and keep `payload` as the preview/list summary source. This separation avoids coupling list truncation behavior to the inspector's full-text display.

## Error Handling

- If a threat lacks miner details, the panel should render clear fallback text instead of crashing.
- If a classification is missing, the panel should show an explicit unknown or pending state.
- If there are no threats, both the table and inspector should render stable empty states.

## Testing Strategy

### Component Tests

Add a focused UI test that verifies:

- the newest threat is selected by default
- clicking a different row updates the selected state
- the inspector renders the selected threat's prompt and classification
- the miner metadata shown in the inspector matches the selected row

### Type and Data Safety

Add or update tests around the dashboard data shape if the frontend test setup already covers static mock-backed components. At minimum, the implementation should keep the data types strict enough that missing `redPrompt`, `blueClassification`, `redMiner`, or `blueMiner` fields are caught during development.

### Manual Verification

Manual verification should confirm that:

- the table remains readable with the new interactive state
- selecting rows updates the detail panel correctly
- the panel shows the full prompt and blue classification for the same threat event
- the layout remains usable on both desktop and mobile widths

## File-Level Impact

Expected files to modify or add:

- `frontend/app/dashboard/page.tsx`
  Own selected-threat state and render the table plus inspector layout.
- `frontend/components/dashboard/ThreatTable.tsx`
  Support row selection and active-row styling.
- `frontend/components/dashboard/ThreatDetailsPanel.tsx`
  New component for prompt, classification, and miner metadata.
- `frontend/data/mock.ts`
  Expand threat entry types and seed data with prompt/classification/miner details.
- frontend dashboard test files near the updated components
  Cover row selection and detail rendering.

## Follow-Up

Once the dashboard is connected to live validator output, the same inspector structure can be fed from real threat events instead of mock data. The current design intentionally keeps that migration straightforward by treating each table row as a self-contained threat record with an attached detail payload.
